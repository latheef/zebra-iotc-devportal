# Step 2: Discover Your Reader

đź“— **TUTORIAL**: In this step, you send a `get_version` command to your reader via the MGMT interface to verify it is online and communicating through the IOTC broker.

## What You Will Do

1. Open a second terminal for publishing commands
2. Publish a `get_version` command to the MGMT command topic
3. Observe the response in your subscriber terminal (from Step 1)

## Publish the `get_version` Command

Open a **new terminal** (keep the subscriber terminal from Step 1 running) and publish the following command:

```bash
mosquitto_pub -h <BROKER_HOST> -p 8883 \
  -u "<MQTT_USERNAME>" -P "<MQTT_PASSWORD>" \
  --capath /etc/ssl/certs \
  -t "<TENANT_ID>/MGMT/clients/cmnd/<DEVICE_SERIAL>" \
  -m '{"command":"get_version","requestId":"1"}'
```

### Command Payload

```json
{
  "command": "get_version",
  "requestId": "1"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `command` | string | The command name to execute on the reader |
| `requestId` | string | A client-generated identifier to correlate requests with responses |

## Expected Response

Switch to your subscriber terminal (from Step 1). You should see a response similar to:

```json
{
  "command": "get_version",
  "requestId": "1",
  "status": "OK",
  "payload": {
    "model": "RFD40",
    "serialNumber": "RFD40-1234567890",
    "firmwareVersion": "3.10.27",
    "bluetoothVersion": "1.2.3",
    "hardwareVersion": "A0"
  }
}
```

### Response Fields

| Field | Description |
|-------|-------------|
| `command` | Echoes the command name from the request |
| `requestId` | Echoes the request ID for correlation |
| `status` | `OK` indicates success; any other value indicates an error |
| `payload.model` | Reader model: `RFD40` or `RFD90` |
| `payload.serialNumber` | Unique device serial number |
| `payload.firmwareVersion` | Current firmware version |
| `payload.bluetoothVersion` | Bluetooth module firmware version |
| `payload.hardwareVersion` | Hardware revision identifier |

## Verifying Success

If you received a response with `"status": "OK"`, your reader is:

## Troubleshooting

| Issue | Possible Cause | Resolution |
|-------|---------------|------------|
| No response received | Reader is offline or not connected to Wi-Fi | Verify the reader's Wi-Fi connection and IOTC broker settings via 123RFID Desktop |
| No response received | Incorrect device serial in topic | Double-check the serial number; it is case-sensitive |
| `"status": "ERROR"` | Command not recognized | Verify firmware version is â‰Ą 3.10.27 |
| Response on wrong topic | Subscribed to the wrong topic | Confirm the topic pattern matches `{tenantId}/MGMT/clients/resp/{deviceSerial}` |

> **Note:** The `requestId` field can be any string value you choose. It is passed through unchanged in the response, allowing you to match responses to requests when sending multiple commands.
