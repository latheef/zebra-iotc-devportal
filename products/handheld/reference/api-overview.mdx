# MQTT API Reference Overview

Complete inventory of MQTT 3.1.1 endpoints for managing and operating Zebra RFD40/RFD90 handheld RFID reader sleds through the IoT Connector.

## Protocol

All communication uses **MQTT 3.1.1** with JSON-encoded payloads. The reader connects to a configured MQTT broker and subscribes to command topics. Applications publish commands and subscribe to response and event topics.

## Topic Pattern

```
{tenantId}/{interface}/clients/{channel}/{deviceSerial}
```

| Segment | Description |
|---|---|
| `tenantId` | Tenant identifier assigned during provisioning |
| `interface` | `MGMT`, `CTRL`, or `DATA` |
| `channel` | `cmnd` (command), `resp` (response), `event`, `data1event`, or `data2event` |
| `deviceSerial` | Device serial number (e.g., `24042AR50033`) |

## Interfaces

| Interface | Purpose | Channels |
|---|---|---|
| **MGMT** | Device management; configuration, networking, firmware, certificates | `cmnd`, `resp`, `event` |
| **CTRL** | RFID operations; operating modes, filters, start/stop | `cmnd`, `resp` |
| **DATA** | Tag data; read events streamed from the reader | `data1event`, `data2event` |

## Channel Directions

| Channel | Direction | Description |
|---|---|---|
| `cmnd` | Application → Reader | Commands sent to the device |
| `resp` | Reader → Application | Responses to commands |
| `event` | Reader → Application | Asynchronous events (heartbeats, alerts) |
| `data1event` / `data2event` | Reader → Application | Tag read data |

## Payload Format

All payloads are JSON objects. Every command includes a `command` field and a `requestId` field. Every response includes a `command` field, `requestId`, and a `statusCode`.

### Request Structure

```json
{
  "command": "<command_name>",
  "requestId": "<unique_id>",
  ...
}
```

### Response Structure

```json
{
  "command": "<command_name>",
  "requestId": "<unique_id>",
  "statusCode": 0,
  "statusMessage": "Success",
  ...
}
```

### Standard Status Codes

| Code | Name | Description |
|---|---|---|
| `0` | Success | Command executed successfully |
| `1` | GeneralError | An unspecified error occurred |
| `2` | InvalidCommand | The command name is not recognized |
| `3` | InvalidParameter | A parameter value is out of range or malformed |
| `4` | InvalidRequestId | The `requestId` field is missing or empty |
| `5` | DeviceBusy | The device is processing another command |
| `10` | OperationFailed | The command was understood but execution failed |
| `15` | NotFound | The requested resource does not exist |
| `16` | ResourceInUse | The resource cannot be modified because it is active |

## Field Table Format

Throughout this reference, field tables use the following columns:

| Column | Description |
|---|---|
| **Field** | JSON path using dot notation (e.g., `ntp.status`) |
| **Type** | `string`, `integer`, `boolean`, `number`, `array`, `object` |
| **Required** | Whether the field must be present in the request |
| **Description** | Meaning and constraints |

## Functional Groups

1. **Device Identity**: `get_version`
2. **Device Health**: `get_status`, `heartBeatEVT`, `alerts`, `alert_short`
3. **Networking**: `get_eth`, `get_wifi`, `set_wifi`, `delete_wifi_profile`
4. **Security**: `get_installed_certificate`, `install_certificate`, `delete_certificate`
5. **Endpoint Management**: `get_endpoint_config`, `config_endpoint`
6. **Configuration**: `get_config`, `set_config`, `config_events`
7. **RFID Operations**: `get_operating_mode`, `set_operating_mode`, `control_operation`, `get_post_filter`, `set_post_filter`, `dataEVT`
8. **Firmware & Lifecycle**: `set_os`, `reboot`, `mqttConnEVT`
