# How to Integrate with Google Cloud IoT

This guide walks through connecting Zebra RFD40/RFD90 reader sleds to Google Cloud using the Pub/Sub MQTT bridge for tag data ingestion and device management.

## Prerequisites

- Google Cloud project with Pub/Sub and IoT APIs enabled
- `gcloud` CLI installed and authenticated
- Reader sleds with firmware version 2.0 or later
- RS256 or ES256 key pair for JWT authentication

## Architecture Overview

Google Cloud's IoT integration uses the Pub/Sub MQTT bridge. Readers connect to Google's MQTT endpoint, authenticate with JWTs, and publish data that flows into Pub/Sub topics for downstream processing.

```
┌──────────┐    MQTT/TLS    ┌──────────────┐   Pub/Sub  ┌──────────────┐
│  Reader   │──────────────►│  GCP MQTT    │───────────►│  Dataflow    │
│  Sled     │               │  Bridge      │            │  BigQuery    │
└──────────┘                └──────────────┘            │  Cloud Fn    │
                                                        └──────────────┘
```

## Step 1: Create a Device Registry

Create a Pub/Sub topic for device telemetry:

```bash
gcloud pubsub topics create zebra-rfid-telemetry
```

Create an IoT device registry (or use Google's current IoT device management approach):

```bash
gcloud iot registries create zebra-rfid-registry \
  --region=us-central1 \
  --event-notification-config=topic=zebra-rfid-telemetry
```

## Step 2: Generate Key Pairs and Register Devices

Generate an RS256 key pair for device authentication:

```bash
openssl genpkey -algorithm RSA -out reader-private.pem -pkeyopt rsa_keygen_bits:2048
openssl rsa -in reader-private.pem -pubout -out reader-public.pem
```

Register the reader as a device:

```bash
gcloud iot devices create rfd40-sn12345 \
  --registry=zebra-rfid-registry \
  --region=us-central1 \
  --public-key=path=reader-public.pem,type=rsa-pem
```

## Step 3: Understand JWT Authentication

Google Cloud's MQTT bridge uses JWT (JSON Web Token) for authentication instead of username/password or X.509 client certificates. The reader (or a provisioning service acting on its behalf) must generate a JWT signed with the device's private key.

JWT payload structure:

```json
{
  "iat": 1715673600,
  "exp": 1715760000,
  "aud": "your-gcp-project-id"
}
```

The JWT is passed as the MQTT password field during connection. JWTs expire and must be refreshed; the reader must disconnect and reconnect with a new JWT before expiration.

> **Warning:** JWT tokens have a maximum lifetime of 24 hours. Implement automatic token refresh and reconnection logic. If the reader firmware does not support JWT generation natively, use a local gateway or bridge to handle authentication.

## Step 4: Configure Topic Mapping

Google Cloud's MQTT bridge expects messages on specific topic paths:

**Telemetry (device-to-cloud):**
```
/devices/{device_id}/events
```

**Device state:**
```
/devices/{device_id}/state
```

**Configuration (cloud-to-device):**
```
/devices/{device_id}/config
```

**Commands (cloud-to-device):**
```
/devices/{device_id}/commands/#
```

Map Zebra reader topics to GCP's expected format:

| Zebra Topic | GCP Topic | Data Type |
|---|---|---|
| `zebra/rfid/{device_id}/tagEvents` | `/devices/{device_id}/events` | Telemetry |
| `zebra/rfid/{device_id}/status` | `/devices/{device_id}/state` | Device state |
| `zebra/rfid/{device_id}/set_config` | `/devices/{device_id}/commands/set_config` | Commands |

Since GCP's topic structure differs from Zebra's, a bridge or gateway is typically required for topic remapping.

## Step 5: Configure the Reader's MQTT Endpoint

Configure the reader to connect to Google's MQTT bridge. The `config_endpoint` payload:

```json
{
  "command_id": "gcp-endpoint-001",
  "config": {
    "mqtt": {
      "broker_host": "mqtt.googleapis.com",
      "broker_port": 8883,
      "tls_enabled": true,
      "tls_ca_path": "/certs/google-roots.pem",
      "client_id": "projects/your-project/locations/us-central1/registries/zebra-rfid-registry/devices/rfd40-sn12345",
      "username": "unused",
      "password": "<JWT_TOKEN>",
      "clean_session": true,
      "keepalive_seconds": 60
    }
  }
}
```

> **Note:** The GCP MQTT client ID must follow the full resource path format. The `username` field is ignored by Google's bridge but must be present; set it to any non-empty string.

## Step 6: Process Data in Pub/Sub

Messages from readers arrive in the configured Pub/Sub topic. Set up subscribers to process the data:

**Cloud Functions subscriber:**

```bash
gcloud functions deploy processTagEvents \
  --trigger-topic=zebra-rfid-telemetry \
  --runtime=python311 \
  --entry-point=process_event
```

**Dataflow pipeline:**

Create a Dataflow job to stream tag events from Pub/Sub to BigQuery for analytics:

```bash
gcloud dataflow jobs run zebra-tag-pipeline \
  --gcs-location=gs://dataflow-templates/pubsub-to-bigquery \
  --parameters inputTopic=projects/your-project/topics/zebra-rfid-telemetry,\
outputTableSpec=your-project:rfid_data.tag_events
```

## Step 7: Verify the Integration

1. Confirm the reader connects to `mqtt.googleapis.com` (check GCP IoT device activity)
2. Verify messages appear in the Pub/Sub topic:

```bash
gcloud pubsub subscriptions pull zebra-rfid-telemetry-sub --auto-ack --limit=5
```

3. Check downstream processing (BigQuery, Cloud Functions logs)

## Troubleshooting

| Symptom | Likely Cause | Resolution |
|---|---|---|
| Connection refused | JWT expired or malformed | Regenerate JWT; verify `aud` matches project ID |
| Connected but no messages in Pub/Sub | Wrong topic path | Ensure publishing to `/devices/{id}/events` |
| Frequent disconnections | JWT expiring during connection | Set JWT expiry to maximum (24h); implement auto-refresh |
| MQTT client ID rejected | Incorrect resource path format | Verify full path: `projects/.../registries/.../devices/...` |

> **Note:** Google Cloud IoT Core was deprecated in 2023, but the Pub/Sub MQTT bridge pattern remains available through partner solutions and open-source MQTT brokers deployed on GCP (e.g., EMQX on GKE). Adapt this guide to your chosen MQTT endpoint on Google Cloud.
