alerts

Detailed, structured alert events published when device conditions transition state or cross monitored thresholds. Covers battery, power, firmware, temperature, and network conditions.

Topic​

Topic
Event{tenantId}/MGMT/clients/event/{deviceSerial}

Direction​

Reader → Application

Payload​

Battery Alert​

{
  "type": "ALERT",
  "timestamp": "2026-04-29T12:33:34.279Z",
  "state": "ONESHOT",
  "id": "BATTERY",
  "priority": "LOW",
  "alertDetails": {
    "batteryAlert": {
      "status": "CHARGING",
      "stateOfHealth": "GOOD",
      "chargePercentage": 100
    }
  }
}

FieldTypeDescription
typestringMessage category: EVENT, NOTIFICATION, or ALERT
timestampstringAlert timestamp
statestringAlert state (see table below)
idstringAlert type identifier (see table below)
prioritystringPriority: CRITICAL, HIGH, MEDIUM, or LOW
alertDetailsobjectDomain-specific details (structure varies by id)

Alert States​

StateDescription
SETAlert condition is currently active
CLEARAlert condition has been resolved
ONESHOTOne-time event — fires once, no corresponding CLEAR

Alert Identifiers​

idstate ValuesTrigger Condition
BATTERYONESHOTBattery charge or health state changes
POWERONESHOTPower source changes (e.g., USB to battery)
NETWORK_EVENTONESHOTWi-Fi or Ethernet interface connects, disconnects, or changes IP
FIRMWARE_UPDATESET / CLEARFirmware update starts (SET) or completes/fails (CLEAR)
TEMPERATURESET / CLEARTemperature crosses threshold (SET) or returns to normal (CLEAR)

Notes​

  • Alerts are enabled and configured via config_events. Each alert type can be independently enabled or disabled.
  • ONESHOT events fire once when the condition occurs and do not require a corresponding CLEAR.
  • SET / CLEAR pairs indicate persistent conditions — SET when active, CLEAR when resolved.
  • Alert events are published on the MGMT event topic alongside heartBeatEVT and mqttConnEVT.
