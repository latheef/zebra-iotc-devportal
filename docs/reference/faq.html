Frequently Asked Questions

📕 REFERENCE: Answers to common questions about the Zebra IoT Connector for handheld RFID readers.

General​

What MQTT version does the IoT Connector support?​

The IoT Connector on RFD40/RFD90 devices supports MQTT 3.1.1 only. MQTT 5.0 is not supported. Ensure your broker and client libraries are configured for MQTT 3.1.1 protocol level.

Can I use a REST API with handheld readers?​

No. The handheld IoT Connector communicates exclusively via MQTT. Unlike the fixed IoT Connector (which offers both REST and MQTT), the handheld reader sleds use MQTT as their sole protocol. All commands, responses, and events are exchanged over MQTT topics.

Which Zebra devices support the handheld IoT Connector?​

The RFD40 and RFD90 RFID reader sleds. These are Bluetooth-paired accessories that attach to compatible Zebra mobile computers (e.g., TC52, TC57, TC72, TC77 series).

What is the relationship between the reader sled and the host mobile device?​

The reader sled connects to a Zebra mobile computer via Bluetooth. The mobile device provides the network connection (Wi-Fi) that the IoT Connector uses to reach the MQTT broker. RFID operations are performed by the reader sled's hardware; the mobile device acts as a network gateway.

Do I need the 123RFID Desktop application?​

Yes, for initial provisioning tasks such as setting the regulatory region, performing firmware updates, and initial Bluetooth pairing. After provisioning, day-to-day operations are managed entirely via MQTT.

Connectivity & Network​

What port does MQTT use?​

Standard MQTT uses port 1883 (unencrypted) and port 8883 (TLS-encrypted). For production deployments, always use port 8883 with TLS enabled.

Can the device connect to multiple MQTT brokers simultaneously?​

No. The device connects to one MQTT broker at a time. You can configure multiple endpoints and switch between them, but only one connection is active.

What happens if the MQTT connection drops?​

The device automatically attempts to reconnect using its configured retry settings. During the disconnection period, RFID operations that are in progress continue running, and data is queued locally on the device. Once reconnected, queued data is published.

How does the device identify itself to the broker?​

The device uses its serial number as the MQTT client ID by default. This ensures uniqueness across your fleet. The client ID can be observed in broker connection logs.

What authentication methods are supported?​

The IoT Connector supports username/password authentication and TLS client certificate authentication. The method depends on your broker configuration and the endpoint settings on the device.

How many Wi-Fi profiles can the device store?​

The device supports a maximum of 10 saved Wi-Fi profiles. If you need to add a new profile when 10 already exist, delete an unused profile first.

Does the device support IPv6?​

The device connects via the host mobile device's network stack. IPv6 support depends on the host device's Wi-Fi configuration and the network infrastructure.

RFID Operations​

How do I set the regulatory region?​

The regulatory region must be set using the 123RFID Desktop application. It cannot be set remotely via MQTT for regulatory compliance reasons. The region must be configured before any RFID operations can be performed.

What is the maximum transmit power?​

The maximum transmit power varies by region and device model. Typical maximum is 30 dBm (300 on the power index scale). Check the regional regulatory information for your operating region.

Can I run inventory continuously?​

Yes, by configuring the trigger mode to IMMEDIATE with no timeout, or by using BARCODE_TRIGGER/RFID_TRIGGER modes for on-demand operation. Be mindful of battery consumption during continuous operation; typical battery life under continuous inventory is 4–6 hours.

What is the difference between Session 0 and Session 1?​

Session 0 causes tags to respond on every interrogation cycle, giving the highest re-read rate. Use it for presence detection and dock-door scenarios. Session 1 uses tag-state persistence, so each tag responds once and then goes silent for a persistence period. Use it for counting unique tags in large populations.

Can I read and write tags?​

Yes. The IoT Connector supports tag read operations (inventory) and tag write operations (writing to EPC, User, or Reserved memory banks via access commands). Write operations require specifying the target tag, memory bank, offset, and data.

What tag standards are supported?​

The devices support EPC Gen2 (ISO 18000-63) UHF RFID tags operating in the 860–960 MHz range (exact band depends on the regulatory region).

How do I filter tags during inventory?​

Configure tag filters in the inventory settings. You can filter by EPC prefix (mask and match), memory bank content, or RSSI threshold. Filters reduce unwanted reads and improve application performance.

What happens if the Bluetooth connection drops during an RFID operation?​

The RFID operation continues running on the reader sled. Tag data is queued locally. When Bluetooth connectivity is restored with the host device and the MQTT connection is re-established, queued tag data is published. No data is lost during brief disconnections.

Fleet Management​

How often should I set the heartbeat interval?​

It depends on your monitoring requirements versus battery life. A 60-second heartbeat is a good balance for most deployments. Shorter intervals (15–30 seconds) provide faster detection of offline devices but increase battery consumption. Longer intervals (120–300 seconds) conserve battery but delay offline detection.

Can I control the reader from multiple MQTT clients simultaneously?​

Yes. The IoT Connector uses a topic-based command model. Any MQTT client that publishes to the device's command topic can send commands. However, there is no built-in locking mechanism, if two clients send conflicting commands (e.g., one starts inventory while another stops it), the device processes them in the order received. Design your architecture to avoid command conflicts, or implement application-level coordination.

How do I update firmware on multiple devices?​

Firmware updates for the handheld readers are performed via the 123RFID Desktop application or Zebra's StageNow tool for mass deployment. Over-the-air firmware updates via MQTT are supported on newer firmware versions; check the firmware changelog for OTA update support.

Can I query the firmware version remotely?​

Yes. Send the get_version command on the device's command topic. The response includes the firmware version, IoT Connector version, and hardware model.

How do I monitor device health across a fleet?​

Subscribe to the heartbeat events on each device's event topic. Heartbeats include battery level, Wi-Fi signal strength, connection uptime, and device status. Implement a server-side monitor that alerts when a device misses its expected heartbeat window.

Migration & Compatibility​

Can I migrate from the RFID SDK to the IoT Connector?​

Yes, but the programming models are different. The RFID SDK is a local API running on the host device, while the IoT Connector is a cloud-native MQTT interface. Migrating requires re-architecting your application to use MQTT pub/sub instead of local method calls. The command and event structures are different from the SDK's API.

Is the MQTT topic structure the same as the fixed IoT Connector?​

The topic structure follows a similar pattern (zebra/iotc/{type}/{deviceId}), but the available commands and event payloads differ between handheld and fixed reader products. You cannot use a fixed reader integration directly with handheld readers without modifications.

Can I use the same MQTT broker for both fixed and handheld readers?​

Yes. Both product lines can connect to the same broker. Use the device ID in the topic to distinguish between devices. The command sets are product-specific, so your application logic should handle each device type appropriately.

What happens if I send a fixed reader command to a handheld device?​

The device returns error code 1 (Invalid Command) for any command it does not recognize. No harm is done, but the command has no effect.

Are there breaking changes between firmware versions?​

Breaking changes are rare and documented in the firmware changelog. Zebra maintains backward compatibility for command and event structures within a major version. Always review the changelog before deploying firmware updates.

Data & Security​

Is data encrypted in transit?​

When TLS is enabled on the MQTT connection (port 8883), all data between the device and the broker is encrypted using TLS 1.2. Data between the broker and your application clients should also use TLS. Data is not encrypted while queued locally on the device during disconnection periods.

How long is data queued on the device during disconnections?​

The device queues data in its internal storage until the MQTT connection is restored. Storage is limited; the device can typically hold several thousand tag events. If storage fills, the oldest undelivered events may be discarded. The storageNearlyFull exception event (code 1405) warns when storage usage exceeds 90%.

Can I use MQTT over WebSockets?​

The device's IoT Connector connects directly via TCP-based MQTT, not WebSockets. However, your MQTT broker may offer a WebSocket listener that your web-based application clients can use. The device-to-broker connection remains standard MQTT over TCP.
