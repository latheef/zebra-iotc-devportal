MQTT Topic Quick Reference

📕 REFERENCE: One-page cheat sheet of all IoT Connector MQTT topics, directions, and purposes.

Topic Pattern​

All IoT Connector topics follow the pattern:

zebra/iotc/{type}/{deviceId}

Where {deviceId} is the device serial number (e.g., RFD401234567890).

Complete Topic Table​

Command & Response Topics​

TopicDirectionQoSPurpose
zebra/iotc/command/{deviceId}Client → Device1Send commands to the device
zebra/iotc/response/{deviceId}Device → Client1Receive command responses

Event Topics​

TopicDirectionQoSPurpose
zebra/iotc/event/{deviceId}Device → Client0/1Tag reads, heartbeats, status changes, exception events

Management Topics​

TopicDirectionQoSPurpose
zebra/iotc/management/command/{deviceId}Client → Device1Fleet management commands (firmware, config push)
zebra/iotc/management/response/{deviceId}Device → Client1Fleet management command responses
zebra/iotc/management/event/{deviceId}Device → Client1Fleet management events (update progress, health reports)

Wildcard Subscriptions​

Use MQTT wildcards to subscribe to multiple devices or topic types:

PatternUse Case
zebra/iotc/event/#All events from all devices
zebra/iotc/event/RFD40+All events from devices with IDs starting with RFD40 (single-level)
zebra/iotc/response/{deviceId}Responses from a specific device
zebra/iotc/+/{deviceId}All topic types for a specific device
zebra/iotc/#Everything (use for debugging only)

Warning: Wildcard subscriptions on zebra/iotc/# can produce high message volumes in production environments. Use targeted subscriptions in production.

Commands Quick Reference​

Commands are sent as JSON payloads to zebra/iotc/command/{deviceId}:

CommandPurposeReturns
get_versionQuery firmware and IoT Connector versionVersion details
get_configRetrieve full device configurationConfiguration document
set_configUpdate device configurationSuccess/error
start_inventoryBegin RFID tag inventorySuccess/error
stop_inventoryStop active RFID inventorySuccess/error
get_inventory_statusCheck if inventory is runningInventory state
read_tagRead specific memory bank of a tagTag data
write_tagWrite data to a tag memory bankSuccess/error
lock_tagLock tag memory bankSuccess/error
kill_tagPermanently disable a tagSuccess/error
create_endpointAdd an MQTT broker endpointSuccess/error
delete_endpointRemove an MQTT broker endpointSuccess/error
connect_endpointSwitch to a different endpointSuccess/error
add_wifi_profileAdd a Wi-Fi network profileSuccess/error
delete_wifi_profileRemove a Wi-Fi profileSuccess/error
connect_wifiConnect to a saved Wi-Fi profileSuccess/error
install_certificateInstall a TLS certificateSuccess/error
delete_certificateRemove a TLS certificateSuccess/error
rebootRestart the deviceSuccess (then disconnect)
set_heartbeatConfigure heartbeat intervalSuccess/error

Event Types Quick Reference​

Events are received on zebra/iotc/event/{deviceId}:

Event TypePayload FieldPurpose
tagReadEVTeventType: "tagReadEVT"Individual or batched tag read data
heartbeatEVTeventType: "heartbeatEVT"Periodic device health and status
inventoryStatusEVTeventType: "inventoryStatusEVT"Inventory operation started/stopped
exceptionEVTeventType: "exceptionEVT"Error and warning conditions
connectionEVTeventType: "connectionEVT"MQTT/Wi-Fi/Bluetooth connection state changes
batteryEVTeventType: "batteryEVT"Battery level and charging status changes

Subscription Checklist​

For a minimal integration, subscribe to:

  1. zebra/iotc/response/{deviceId}: to receive command responses
  2. zebra/iotc/event/{deviceId}: to receive tag data and device events

For fleet management, also subscribe to:

  1. zebra/iotc/management/response/{deviceId}: management command responses
  2. zebra/iotc/management/event/{deviceId}: management events

QoS Recommendations​

Use CaseRecommended QoS
Commands and responsesQoS 1 (at least once)
Tag read events (high throughput)QoS 0 (at most once)
Exception eventsQoS 1 (at least once)
Heartbeat eventsQoS 0 (at most once)
Fleet managementQoS 1 (at least once)

Note: QoS 2 (exactly once) is not commonly used with the IoT Connector. QoS 1 provides a good balance between reliability and throughput. Use QoS 0 for high-volume tag events where occasional message loss is acceptable.
