About the Interface Model: MGMT, CTRL, DATA, MDM

📘 EXPLANATION

Overview​

IOTC organizes all MQTT communication into four logical interfaces. Each interface serves a distinct purpose, carries different payload types, and follows its own topic naming convention. This separation of concerns simplifies client implementation, enables fine-grained access control, and allows independent scaling of data streams.

Topic Pattern​

All IOTC topics follow a consistent five-segment pattern:

{tenantId}/{interface}/clients/{channel}/{deviceSerial}
SegmentDescriptionExample
{tenantId}Organization or deployment identifieracme-corp
{interface}Interface name: mgmt, ctrl, data, or mdmmgmt
clientsFixed literal segmentclients
{channel}Logical client grouping identifierwarehouse-app
{deviceSerial}Reader sled serial number23044501500123

Additional sub-segments are appended depending on the message direction:

DirectionSub-segmentExample
Command (app → reader)/commandacme-corp/mgmt/clients/wh-app/23044501500123/command
Response (reader → app)/responseacme-corp/mgmt/clients/wh-app/23044501500123/response
Event (reader → app)/eventacme-corp/mgmt/clients/wh-app/23044501500123/event
Data (reader → app)(none)acme-corp/data/clients/wh-app/23044501500123

Interface Summary​

InterfaceTopic SegmentDirectionQoSPurpose
MGMTmgmtBidirectionalQoS 1Device configuration, health, firmware, operating mode
CTRLctrlBidirectionalQoS 1Operational commands; start/stop inventory, set RF power
DATAdataReader → AppQoS 0Tag observation streaming
MDMmdmBidirectionalQoS 1MDM platform integration (SOTI Connect)

MGMT Interface (Management)​

The Management interface handles all device lifecycle and configuration operations. It is the most feature-rich interface, encompassing device discovery, configuration management, health monitoring, and firmware operations.

Command Categories​

CategoryCommandsDescription
Device Infoget_device_info, get_capabilitiesQuery model, serial, firmware version, supported features
Configurationget_config, set_config, set_config_bulk, reset_configRead/write reader parameters (RF power, session, filters)
Healthget_health, get_battery_statusQuery temperature, battery, uptime, error counters
Operating Modeset_operating_mode, get_operating_modeSwitch between INVENTORY, IDLE, STANDBY modes
Firmwareget_firmware_version, check_firmware_updateQuery and manage firmware lifecycle

Topic Examples​

# Application sends a command
acme-corp/mgmt/clients/warehouse-app/23044501500123/command

# Reader publishes response
acme-corp/mgmt/clients/warehouse-app/23044501500123/response

# Reader publishes asynchronous events
acme-corp/mgmt/clients/warehouse-app/23044501500123/event

CTRL Interface (Control)​

The Control interface provides real-time operational commands for RFID read/write operations. These commands directly affect the reader's RF behavior and are typically latency-sensitive.

Command Categories​

CategoryCommandsDescription
Inventorystart_inventory, stop_inventory, get_inventory_statusStart/stop/query active tag reading
Tag Accessread_tag_memory, write_tag_memory, lock_tag, kill_tagRead/write specific tag memory banks
Antennaset_antenna_config, get_antenna_configConfigure antenna port parameters
Calibrationrun_antenna_checkTest antenna connectivity and impedance

Topic Examples​

# Application sends an operational command
acme-corp/ctrl/clients/warehouse-app/23044501500123/command

# Reader publishes operational response
acme-corp/ctrl/clients/warehouse-app/23044501500123/response

DATA Interface​

The Data interface is a unidirectional, high-throughput stream carrying tag observation data from the reader to subscribing applications. Unlike MGMT and CTRL, there is no command sub-topic; the application only subscribes.

Payload Structure​

Each DATA message contains a batch of one or more tag observations with fields including EPC, RSSI, timestamp, antenna port, phase angle, channel index, and read count.

Topic Example​

# Application subscribes to tag data
acme-corp/data/clients/warehouse-app/23044501500123

QoS and Delivery​

DATA messages use QoS 0 (at most once) to maximize throughput and minimize latency. In high-volume tag reading scenarios, occasional message loss is acceptable; duplicate tag reads in subsequent batches provide natural redundancy.

MDM Interface​

The MDM interface is reserved for integration with enterprise Mobile Device Management platforms. It follows the same topic pattern but uses the mdm segment.

Capabilities​

OperationDescription
Device enrollmentRegister a reader with the MDM platform
Policy enforcementApply configuration policies to reader groups
Firmware deploymentPush firmware updates to managed readers
Compliance checkVerify reader configuration against policy

Topic Examples​

# MDM platform sends a management command
acme-corp/mdm/clients/soti-connect/23044501500123/command

# Reader publishes MDM response
acme-corp/mdm/clients/soti-connect/23044501500123/response

Wildcard Subscriptions​

MQTT wildcard characters enable subscribing to multiple readers or interfaces simultaneously:

WildcardPatternEffect
+ (single-level)acme-corp/data/clients/wh-app/+Subscribe to data from all readers on the wh-app channel
# (multi-level)acme-corp/mgmt/#Subscribe to all MGMT traffic for the entire tenant
Combinedacme-corp/+/clients/wh-app/+/eventSubscribe to all events from all interfaces, all readers
Caution

Wildcard subscriptions to high-volume topics (especially DATA with #) can overwhelm client applications in large fleet deployments. Use targeted subscriptions in production and reserve wildcards for development and debugging.
