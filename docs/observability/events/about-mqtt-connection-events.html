About MQTT Connection Events

MQTT connection events (mqttConnEVT) report changes in the reader's MQTT connection state. These events enable your application to track when devices come online, go offline, or experience reconnection cycles — critical information for fleet connectivity management.

Connection States​

The mqttConnEVT reports three connection states:

CONNECTED​

The device has successfully established an MQTT session with the broker. This event fires:

  • On initial connection after power-on
  • After a successful reconnection following a disconnection
  • After a network recovery
{
  "event_type": "mqttConnEVT",
  "device_serial": "RFD401234567890",
  "timestamp": "2026-05-14T08:00:00.000Z",
  "connection_state": "CONNECTED",
  "broker_url": "mqtts://broker.example.com:8883",
  "disconnect_reason": null,
  "session_id": "sess-abc123"
}

DISCONNECTED​

The device has lost its MQTT connection. The disconnect_reason field indicates why:

ReasonDescription
NETWORK_LOSSWi-Fi or network connectivity lost
BROKER_UNAVAILABLEBroker rejected the connection or is down
KEEPALIVE_TIMEOUTMQTT keepalive ping was not acknowledged
CLIENT_DISCONNECTDevice intentionally disconnected (shutdown, sleep)
SESSION_EXPIREDBroker expired the session
{
  "event_type": "mqttConnEVT",
  "device_serial": "RFD401234567890",
  "timestamp": "2026-05-14T10:30:00.000Z",
  "connection_state": "DISCONNECTED",
  "broker_url": "mqtts://broker.example.com:8883",
  "disconnect_reason": "NETWORK_LOSS",
  "session_id": "sess-abc123"
}

RECONNECTING​

The device is attempting to re-establish the MQTT connection. This state indicates the device is aware of the disconnection and actively retrying.

{
  "event_type": "mqttConnEVT",
  "device_serial": "RFD401234567890",
  "timestamp": "2026-05-14T10:30:05.000Z",
  "connection_state": "RECONNECTING",
  "broker_url": "mqtts://broker.example.com:8883",
  "disconnect_reason": "NETWORK_LOSS",
  "session_id": "sess-abc123"
}

Note: The RECONNECTING event is published if the device has a secondary communication path or if the event was queued before full disconnection. In a complete network outage, this event may not be received until the device reconnects.

How Connection Events Are Delivered​

Connection events have a delivery challenge: a DISCONNECTED event needs to be sent over the connection that just failed. The RFD40/RFD90 handles this through two mechanisms:

  1. Pre-disconnect publish: If the device detects an impending disconnection (e.g., Wi-Fi signal dropping), it publishes the event before the connection is fully lost
  2. MQTT Last Will and Testament (LWT): The device registers an LWT message with the broker at connection time. If the broker detects an ungraceful disconnection, it publishes the LWT message on behalf of the device

For planned disconnections (CLIENT_DISCONNECT), the device publishes the event directly before closing the connection.

Fleet Dashboard Use Cases​

Online/Offline Tracking​

Maintain a device status table updated by connection events:

Device SerialLast StateSinceDisconnect Reason
RFD401234567890CONNECTED08:00:00N/A
RFD409876543210DISCONNECTED10:30:00NETWORK_LOSS
RFD405555555555RECONNECTING10:30:05KEEPALIVE_TIMEOUT

Connectivity Trend Analysis​

Track reconnection frequency per device over time. Devices with frequent reconnection cycles may indicate:

  • Poor Wi-Fi coverage in their operating area
  • Broker capacity issues
  • Network infrastructure problems
  • Device hardware issues (failing Wi-Fi radio)

Shift Coverage​

Monitor connection events at shift start times to verify all devices are online and operational before the work begins. A device that does not send a CONNECTED event during the expected startup window needs attention.

Subscribing to Connection Events​

Single device:

{tenantId}/DATA/clients/connection/{deviceSerial}

All devices:

{tenantId}/DATA/clients/connection/+

Connection events are published at QoS 1 for reliable delivery.

Combining with Heartbeats​

Connection events capture state transitions, while heartbeats confirm ongoing connectivity. Use both together:

  • mqttConnEVT tells you when a device connected or disconnected
  • heartBeatEVT tells you the device is still connected and healthy

If heartbeats stop but no DISCONNECTED event was received, the device may have lost connectivity abruptly (power failure, hardware crash) before the LWT could fire.
