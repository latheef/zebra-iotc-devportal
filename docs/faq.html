Frequently Asked Questions

This section collects the questions that surface after the first week of real integration work; the edge cases, trade-off explanations, and "why does it work this way?" moments that the tutorials and API references do not address directly. If your question is not here, the Troubleshooting Guide (Chapter 14) covers symptom-based diagnosis.

15.1 General​

Why does the reader need Bluetooth and Wi-Fi? Can't it just connect directly to MQTT over Wi-Fi?​

The RFID radio hardware and the IoT Connector agent are separate subsystems. The RFID radio (the component that transmits RF energy, powers tags, and decodes backscattered signals) communicates with its host processor via Bluetooth. The IoT Connector agent runs on the host processor and uses Wi-Fi to reach the MQTT broker. These are not redundant connections; they serve different physical layers. Bluetooth carries the raw RF data between the radio and the processor. Wi-Fi carries the structured MQTT messages between the processor and your infrastructure. If Bluetooth drops, MQTT still works but no tag data flows. If Wi-Fi drops, the radio still scans but data cannot reach the broker. The get_status response distinguishes these states: radioConnection reflects Bluetooth, while the presence of MQTT responses at all confirms Wi-Fi connectivity. See Chapter 14, Section 14.2 for diagnosing each path independently.

Can I use WSS (WebSockets) instead of raw MQTT to connect readers to our existing infrastructure?​

No. The reader's IoT Connector firmware implements MQTT 3.1.1 over TCP/TLS; it does not support MQTT over WebSockets. This is a firmware constraint, not a protocol limitation. If your backend infrastructure requires WebSocket access to MQTT data (e.g., browser-based dashboards), place a protocol bridge between the broker and your frontend. Most enterprise brokers (HiveMQ, EMQX) and cloud platforms (AWS IoT Core) natively support WebSocket listeners alongside standard MQTT listeners, so the broker can accept raw MQTT from readers and serve WebSocket connections to browser clients simultaneously. No data translation is required; the bridge operates at the transport layer, not the message layer.

What is the minimum firmware version required for the MQTT API?​

The MQTT API documented here requires IoT Connector firmware 3.10.27 or later. Earlier firmware versions may support a subset of commands but do not implement the full V1.1 API surface (28 endpoints), and response payload schemas may differ. Verify the running firmware via get_version; the cloudAgentVersion field identifies the IoT Connector agent, and firmwareVersion identifies the reader's overall firmware. If you encounter commands returning response code 2 (invalid payload) with payloads that match the documented schema, a firmware version mismatch is the most likely cause. See the get_version API Reference for field definitions.

Does the reader buffer data locally when it loses MQTT connectivity?​

The IoT Connector agent maintains a limited local buffer for outbound messages. If Wi-Fi drops during an active inventory, the reader continues scanning and buffers tag data. When connectivity resumes, buffered messages are published to the broker. However, the buffer is finite, under sustained high-throughput scanning (hundreds of tags per second), it fills within seconds to minutes depending on the read rate. Once the buffer is full, the oldest messages are dropped (FIFO eviction). For QoS 0 tag data, this means some reads are permanently lost during the outage. For QoS 1 management messages (heartbeats, alerts), the broker's persistent session will also hold undelivered messages, provided cleanSession is set to false on the reader's endpoint configuration. See Chapter 13, Section 13.4 for session persistence configuration guidance.

15.2 MQTT and Connectivity​

Why does my reader disconnect from the broker when I start a high-speed inventory?​

It probably is not disconnecting; the broker is declaring it disconnected because the reader missed its keepalive deadline. During a high-speed inventory, the reader's processor is saturated with RF data processing, serial communication over Bluetooth, JSON serialization, and concurrent Wi-Fi transmission of dataEVT messages. If the MQTT keepalive interval is set too aggressively (e.g., 15 seconds), the keepalive ping may be delayed past 1.5× the interval (the MQTT 3.1.1 spec allows the broker to disconnect after this threshold), triggering a KEEPALIVE_TIMEOUT disconnection. Fix: Increase the keepalive interval to 60 seconds via config_endpoint. This gives the reader sufficient headroom to service the keepalive timer even under peak RF load. Do not set it higher than 120 seconds; longer intervals delay offline detection. See Chapter 10 for keepalive and heartbeat interval tuning guidance.

What happens if two applications subscribe to the same reader's response topic?​

Both receive every response. MQTT pub/sub delivers messages to all subscribers on a topic; there is no exclusive consumer model in standard MQTT 3.1.1. If Application A publishes a get_status command and Application B is also subscribed to <tenant>/MGMT/clients/resp/<serial>, both A and B receive the response. This is usually harmless for read-only commands, but it creates a race condition for stateful operations: if both applications send set_operating_mode, the reader processes them sequentially and the last one wins, but both applications receive both responses, and correlating which response belongs to which application requires careful requestId management. Best practice: Use unique, application-scoped requestId prefixes (e.g., app-a-001, app-b-001) so each application can filter responses by its own prefix and ignore responses to commands it did not send. If you need exclusive consumer semantics for tag data, use shared subscriptions on an enterprise broker (Chapter 13, Section 13.3).

Why do I see mqttConnEVT with reason BROKER_UNAVAILABLE even though the broker is running?​

The BROKER_UNAVAILABLE reason in mqttConnEVT does not mean the broker process is down; it means the reader's TLS handshake or MQTT CONNECT packet was rejected. The most common causes: (1) the broker's listener reached its maximum connection limit and refused the new connection, (2) the reader's client certificate expired or does not chain to a CA in the broker's trust store, or (3) the broker's server certificate changed (e.g., after a cert rotation) and the reader's installed CA no longer validates it. The name is misleading; BROKER_UNAVAILABLE means "the reader tried to connect and failed," not "the broker is offline." Diagnose by checking broker-side connection logs for the specific rejection reason. See Chapter 14, Section 14.1 for the TLS handshake failure diagnostic sequence.

Can I publish commands to multiple readers simultaneously, or must I wait for each response?​

You can publish commands to multiple readers simultaneously. Each reader has its own cmnd topic (<tenant>/<interface>/clients/cmnd/<serial>), and the broker routes each message independently. The readers process commands in parallel; there is no global serialization. However, within a single reader, commands on the same interface are processed sequentially: if you send two MGMT commands to the same reader in rapid succession, the second queues behind the first. Commands on different interfaces (e.g., MGMT and CTRL) are processed on separate connections and do not queue behind each other. For fleet-wide operations (firmware push, configuration update), publish commands to all target readers as fast as your MQTT client allows; the broker and readers handle the fan-out. Monitor per-account publish rate limits if using a cloud broker (Chapter 13, Section 13.1).

15.3 RFID Operations​

Why do I get duplicate tag reads even after enabling post-filters?​

Because post-filters and deduplication solve different problems. A post-filter (set_post_filter) suppresses dataEVT messages based on matching criteria; for example, "only report tags whose EPC starts with 3034." It does not deduplicate. If a tag matching the filter is read 50 times per second, all 50 reads pass the filter and generate 50 dataEVT messages. The deduplication you likely want is a unique tag reporting interval; configured via reportUniqueTags in the operating mode, which suppresses re-reports of the same EPC within a time window. Alternatively, implement deduplication in your subscriber application using the epc + timestamp fields in dataEVT payloads. For edge-side deduplication strategies, see Chapter 9, Section 9.2. For post-filter configuration, see Chapter 8, Section 8.5.

I set tagPopulation to 1000 but I only have 200 tags. Does overestimating hurt performance?​

Slightly, and it is almost always the right trade-off. The Gen2 anti-collision algorithm allocates $2^Q$ time slots per inventory round, where Q is derived from tagPopulation. Setting tagPopulation to 1000 when 200 tags are present means Q starts at ~10 (1024 slots) instead of the optimal ~8 (256 slots). The excess 768 empty slots are traversed quickly, each empty slot takes only ~400 µs, adding roughly 0.3 seconds per inventory round. By contrast, underestimating (setting tagPopulation to 50 for 200 tags) causes massive slot collisions that can lose 20–30% of reads per cycle and take multiple rounds to recover. The cost of overestimation is a fraction of a second. The cost of underestimation is missing tags. Always round up. See Chapter 8, Section 8.2 for the Q algorithm explanation.

Why does control_operation START succeed but the reader reports zero tags in an area where I know tags exist?​

This is the most common "everything looks right but nothing works" scenario, and the cause is almost always one of four things checked in this order:

  1. Radio not connected: get_status shows radioConnection: "DISCONNECTED". The Bluetooth link between the IoT Connector and the RFID radio is down. Re-pair the devices.
  2. Region not set: But wait, you said START returned code 0, not code 6? Correct; verify anyway with get_current_region. In rare firmware edge cases, the region state can be inconsistent.
  3. Post-filter suppressing everything: get_post_filter reveals an enabled filter with criteria that no tags in the read zone match. Disable it temporarily to test.
  4. Operating mode set to ADVANCED with no advancedConfigurations: The radio is "running" but has no RF parameters to execute. Switch to a preset profile like FAST_READ to confirm the radio and antenna are functional, then reconfigure ADVANCED mode.

The full diagnostic sequence is in Chapter 14, Section 14.6, Sequence 2.

Can I run inventory on the reader continuously (24/7), or is there a duty cycle limit?​

There is no firmware-imposed duty cycle limit; you can run control_operation START and leave the reader scanning indefinitely. However, three practical constraints apply. First, battery life: continuous scanning at maximum transmit power drains the RFD40's 7000 mAh battery in approximately 4–6 hours, depending on the tag population and RF environment. For 24/7 operation, the reader must be externally powered (cradle, DC, or USB). Second, thermal throttling: sustained RF transmission in high-ambient-temperature environments (> 40°C) may cause the reader to reduce transmit power or pause scanning to stay within its thermal envelope. Monitor temperature in heartbeat events. Third, regulatory duty cycle: in ETSI regions (Europe), Listen Before Talk (LBT) regulations limit how long a reader can transmit on a single channel. The reader firmware handles LBT compliance automatically; you do not need to manage it, but effective throughput is lower than in FCC regions where frequency hopping has no per-channel dwell limit.

15.4 Security and Certificates​

Why does install_certificate return code 0 (success) but the reader still won't connect using the new certificate?​

Because installing a certificate and using a certificate are separate actions. install_certificate stores the certificate in the reader's credential store; it does not automatically bind it to an MQTT endpoint. After installing a new MQTT CA certificate, you must reconfigure the endpoint (via config_endpoint) to use it, or, if you are replacing an existing certificate; the new certificate must share the same name as the one the endpoint is already configured to use. Verify the binding by issuing get_endpoint_config and confirming the endpoint's TLS configuration references the correct certificate name. A related gotcha: if you install a certificate with type: "wifi", it will not be used for any MQTT operation regardless of its content, because the reader selects certificates by type during the TLS handshake. See Chapter 7, Section 7.2 for the certificate installation and binding procedure.

Can I use a self-signed certificate for MQTT TLS in production?​

Technically yes; the reader does not distinguish between self-signed CA certificates and certificates issued by a public CA. If you install your self-signed CA via install_certificate with type: "mqtt", the reader will trust any broker certificate signed by that CA. However, self-signed certificates introduce operational risk at fleet scale. You become your own CA — responsible for certificate issuance, revocation, expiration tracking, and the entire rotation lifecycle across every reader and every broker instance. If your self-signed CA's private key is compromised, there is no external revocation mechanism (no CRL or OCSP responder); you must manually push a new CA to every reader. For deployments under 50 readers in controlled environments, self-signed CAs are pragmatic. For larger fleets, use an intermediate CA issued by your organization's PKI or a managed certificate service. For the rotation procedure regardless of CA type, see Chapter 12, Section 12.2.

The reader has three certificate types: mqtt, wifi, and client. When do I need all three?​

Only when you are using mutual TLS for MQTT and 802.1X (EAP-TLS) for Wi-Fi authentication. Here is when each type is required:

  • mqtt: Always required when connecting to a TLS-enabled broker. This is the CA certificate that the reader uses to verify the broker's server certificate. Without it, the TLS handshake fails.
  • client: Required only when the broker demands mutual TLS (mTLS). This is the reader's own certificate and private key, presented to the broker during the handshake to prove the reader's identity. Many deployments use username/password or Thing-level authentication (AWS IoT Core) instead, in which case no client certificate is needed on the reader side.
  • wifi: Required only for WPA2/WPA3-Enterprise with EAP-TLS authentication. If your Wi-Fi network uses WPA2-Personal (PSK) or WPA2-Enterprise with PEAP (username/password), no wifi certificate is needed.

Most deployments need mqtt only. Deployments with mTLS need mqtt + client. Deployments with both mTLS and EAP-TLS Wi-Fi need all three. See Chapter 7 for each configuration path.

15.5 Fleet Management​

I pushed a firmware update to 500 readers simultaneously and now 30% are offline. What happened?​

You skipped the canary deployment. When set_os is sent to all readers at once, every reader downloads the firmware package, validates it, reboots, and reconnects within a 60–90 second window. If the firmware has any issue (a TLS library change that is incompatible with your broker's cipher suite configuration, a default endpoint reset that clears the reader's MQTT credentials, or a boot-time initialization that requires longer than the broker's keepalive timeout), the failure affects every reader simultaneously. The 30% that are offline likely share a common characteristic (same model, same broker endpoint, same certificate type) that interacts badly with the new firmware. Recovery: For readers that are still online, issue get_version to confirm their firmware. For offline readers, connect via USB/Bluetooth with 123RFID Desktop to diagnose. Prevention: Always use staged rollouts: 1% canary, 5%, 25%, 100%, with soak periods between stages. See Chapter 12, Section 12.1 for the full canary deployment procedure.

How do I know when a reader's certificate is about to expire if I have 500 readers?​

You do not monitor individual certificates; you build a fleet-wide expiration report. Run get_installed_certificate against every reader on a weekly schedule (via your fleet management application) and extract the validTo field from each certificate. Aggregate the results into an expiration timeline sorted by earliest expiry. Set alert thresholds at 90 days (plan rotation), 30 days (begin rotation), and 7 days (critical; expedite). The get_config command returns all installed certificates in a single response if you prefer a comprehensive snapshot per reader. Certificate expiration is the most preventable cause of fleet-wide outages, and the most common. See Chapter 12, Section 12.2 for the monitoring schedule and zero-downtime rotation procedure.

Can I use get_config to snapshot a "golden" reader and clone its configuration to new readers?​

Yes; this is the recommended provisioning pattern. Issue get_config against a fully configured, validated reader to capture its complete state: endpoint configuration, Wi-Fi profiles, installed certificates (metadata only; PEM content is not returned), event configuration, and operating mode. Use this snapshot as a template to drive config_endpoint, set_wifi, install_certificate, config_events, and set_operating_mode commands against new readers in your provisioning pipeline. Two caveats: (1) get_config does not return certificate PEM content or Wi-Fi passwords; you must supply these from your secrets management system during provisioning. (2) The deviceSerial in the snapshot is the source reader's serial; do not blindly replay the full response as commands; extract the configuration values and apply them with appropriate per-device adjustments (e.g., tenantId may vary by site). For the full golden-configuration workflow, see Chapter 11, Section 11.2.

Why do heartbeat events stop arriving when many readers are simultaneously running inventory?​

The broker or your subscriber is saturated with tag data. During fleet-wide inventory, each reader publishes 10–50 dataEVT messages per second. With 200 active readers, that is 2,000–10,000 messages per second competing for broker processing and subscriber throughput. Heartbeat events (typically one per reader per 30–60 seconds) are low-priority in this flood. On a single-broker architecture, QoS 0 tag data and QoS 1 heartbeats share the same processing pipeline, and the sheer volume of data traffic delays heartbeat delivery beyond your monitoring threshold, causing false "stale" or "offline" classifications. Fixes: (1) Stagger inventory start times using radioStartConditions.startDelay so readers are not all publishing simultaneously. (2) Enable edge-side post-filters to reduce per-reader data volume before it reaches the broker (Chapter 8, Section 8.5). (3) Separate MGMT and DATA traffic onto different brokers so heartbeats are never queued behind tag data (Chapter 13, Section 13.4).
