17.1 Response Codes Master Reference​

Every command response from Zebra RFID Connect includes a response.code integer and a response.description string. The table below lists all response codes, their meanings, the commands or domains where each code may appear, and the recommended developer action. Response codes 11 and 22 have different meanings depending on the domain.

CodeNameDomainDescriptionRecommended Action
0SuccessAllThe command completed successfully.None required.
1AcceptedManagementThe command was accepted and will be processed asynchronously (e.g., firmware update initiated).Subscribe to the corresponding event topic to monitor progress.
2Invalid payloadAllMalformed JSON, missing required fields, or a field value that does not match the expected type.Validate the JSON payload against the command schema. Check for typos, missing commas, or incorrect data types.
3Unable to retrieveManagementThe requested configuration or status could not be read from the device.Retry after a brief delay. If persistent, reboot the reader with reboot.
4Firmware update in progressManagementA firmware update is currently running. The device cannot accept configuration changes until the update completes.Wait for the firmwareUpdateEVT event with a terminal status before retrying.
6Region not configuredControlThe reader's regulatory region has not been set. RFID operations require a valid region.Set the region using the Zebra reader configuration utility or 123RFID Desktop before issuing RFID commands.
10Configuration already existsManagementAn endpoint or configuration with the specified identifier already exists (returned on ADD operations).Use MODIFY instead of ADD, or delete the existing configuration first.
11Inventory in progressControlAn RFID inventory or barcode scan is currently active. The command cannot be executed while an operation is running.Send control_operation with operation: "STOP" before retrying.
11Configuration does not existManagementNo endpoint or configuration matches the specified identifier (returned on MODIFY/DELETE operations).Verify the configuration name with the corresponding get_* command.
12No operation in progressControlA STOP command was sent but no RFID or scan operation is currently running.No action required. This is informational; the reader is already idle.
15SSID not foundManagementThe specified Wi-Fi network SSID was not detected by the reader's radio scan.Verify the SSID is correct and the access point is within range. Retry after ensuring the AP is broadcasting.
17SSID missingManagementThe ssid field is absent from a set_wifi command payload.Add the required ssid field to the payload.
18Certificate parse errorManagementThe uploaded certificate or private key could not be parsed. The file may be corrupted, in an unsupported format, or the PEM encoding is invalid.Verify the certificate is PEM-encoded and includes both BEGIN and END markers. Check for truncation or encoding issues.
19Download failedManagementA file download (firmware image or certificate) failed due to a network error, unreachable URL, or HTTP error.Verify the download URL is accessible from the reader's network. Check firewall rules and DNS resolution.
20Firmware validation failedManagementThe downloaded firmware image failed integrity validation. The file may be corrupted or not intended for this reader model.Re-download the firmware package and verify its checksum against the Zebra support site. Confirm the firmware file matches the target reader model.
21Insufficient batteryManagementThe reader's battery level is too low to perform the requested operation (e.g., firmware update requires ≥30% battery).Connect the reader to a power source or charge the battery before retrying.
22Operation in progressManagementA blocking operation (reboot, OS update) is currently executing.Wait for the operation to complete before sending new commands.
22Advanced config not setControlAn RFID command references ADVANCED operating mode parameters, but set_operating_mode has not been called with profileName: "ADVANCED".Send set_operating_mode with profileName: "ADVANCED" and the required RF parameters before retrying.
23Invalid enum valueAllA field that expects an enumerated value received an unrecognized string.Check the field value against the Enum Values Reference (§17.2). Enum values are case-sensitive.
24Maximum pre-filters exceededControlThe number of pre-filter entries in the select array exceeds the reader's maximum (typically 8).Reduce the number of pre-filter entries. Combine criteria where possible or use post-filters for additional filtering.
28Pattern too longControlA post-filter pattern string exceeds the 64-character maximum.Shorten the pattern. For complex matching, consider splitting into multiple filters or refactoring the regex.

Tip: The response.description string returned by the device is human-readable and may include additional context not shown in this table. Always log it for troubleshooting.

17.2 Enum Values Reference​

This section documents every enumerated value accepted by the Zebra RFID Connect API. All enum values are case-sensitive strings unless otherwise noted. Values are grouped by functional domain.

17.2.1 Operating Mode: profileName​

Used in: set_operating_mode

ValueDescriptionDefault
BALANCED_PERFORMANCEGeneral-purpose profile balancing read rate, range, and battery life. Suitable for most warehouse and retail use cases.✓
FAST_READMaximizes read rate and range. Higher power consumption. Optimized for high-speed conveyor or dock-door applications.
CYCLE_COUNTOptimized for counting unique tags in a defined area. Reduces duplicate reads.
DENSE_READERSMitigates reader-to-reader interference in environments with multiple overlapping readers. Uses spectral management techniques.
OPTIMAL_BATTERYMinimizes power consumption at the cost of read rate and range. Designed for extended handheld shifts without charging.
ADVANCEDUnlocks manual control of all RF parameters (link profile, session, transmit power, Q algorithm, etc.). Requires explicit RF parameter configuration.

17.3 Topic Registry​

Every MQTT message in Zebra RFID Connect is published to or received from a topic that follows the pattern:

{tenantId}/{interface}/clients/{channel}/{deviceSerial}

The table below lists every topic, organized by interface. Direction indicates whether the topic carries cloud-to-device (C2D) or device-to-cloud (D2C) messages.

17.3.1 Management Interface (MGMT)​

Topic PatternChannelDirectionQoSPurpose
{tenantId}/MGMT/clients/cmnd/{deviceSerial}cmndC2D1Send management commands (config_endpoint, config_events, set_wifi, set_certificate, reboot, firmware_update, etc.)
{tenantId}/MGMT/clients/resp/{deviceSerial}respD2C1Receive responses to management commands
{tenantId}/MGMT/clients/event/{deviceSerial}eventD2C1Receive asynchronous management events (heartBeatEVT, alertEVT, exceptionEVT, mqttConnectionEVT, firmwareUpdateEVT)

17.3.2 Control Interface (CTRL)​

Topic PatternChannelDirectionQoSPurpose
{tenantId}/CTRL/clients/cmnd/{deviceSerial}cmndC2D1Send control commands (control_operation, set_operating_mode, set_access_operation, set_post_filter, get_operating_mode, get_post_filter)
{tenantId}/CTRL/clients/resp/{deviceSerial}respD2C1Receive responses to control commands

17.3.3 Data Interface (DATA)​

Topic PatternChannelDirectionQoSPurpose
{tenantId}/DATA/clients/data1event/{deviceSerial}data1eventD2C0Receive RFID tag read events (dataEVT). High-frequency stream during active inventory.
{tenantId}/DATA/clients/data2event/{deviceSerial}data2eventD2C0Receive barcode scan events. Published when the scanner reads a barcode.
{tenantId}/DATA/clients/rfid/{deviceSerial}rfidD2C1Receive RFID access operation results (read, write, lock, kill). Published with the retained flag.

17.3.4 Mobile Device Management Interface (MDM)​

Topic PatternChannelDirectionQoSPurpose
{tenantId}/MDM/clients/cmnd/{deviceSerial}cmndC2D1Send MDM provisioning commands
{tenantId}/MDM/clients/resp/{deviceSerial}respD2C1Receive responses to MDM commands

Topic Naming Summary​

ComponentSourceExample
{tenantId}Assigned during provisioning. Identifies the organization or deployment.acme-corp
{interface}Fixed per endpoint: MGMT, CTRL, DATA, MDMMGMT
{channel}Fixed per direction: cmnd, resp, event, data1event, data2event, rfidcmnd
{deviceSerial}The serial number of the physical reader deviceRFD401234567890

Wildcard subscriptions: To monitor all devices for a tenant, subscribe with a wildcard: {tenantId}/MGMT/clients/event/+ (single-level) or {tenantId}/MGMT/clients/# (multi-level). Use wildcards judiciously; high-volume data topics (data1event) can saturate client buffers.

17.4 RFID Glossary​

Terms are defined in the context of Zebra RFID Connect and the IoT Connector API. Cross-references to relevant chapters and API commands are noted where applicable.

TermDefinition
Access operationA targeted operation on a specific tag's memory: read, write, lock, or kill. Configured via set_access_operation. See §8.4.
Access passwordA 32-bit password stored in the tag's Reserved memory bank. Required to perform lock or kill operations on password-protected tags.
AntennaThe RF radiating element that transmits and receives signals to/from RFID tags. The RFD40 and RFD90 readers have integrated antennas.
BackscatterThe method by which a passive RFID tag communicates with the reader. The tag modulates the reader's transmitted RF signal to encode data.
ChannelThe specific topic segment in the Zebra RFID Connect topic hierarchy that identifies the message direction (e.g., cmnd, resp, event). See §3.2.
Clean sessionAn MQTT connection parameter (cleanSession: true) that instructs the broker to discard any stored session state (subscriptions, queued messages) for the client.
Client IDThe unique MQTT client identifier used by the reader when connecting to the broker. Typically derived from the device serial number.
CommandA C2D (cloud-to-device) JSON message published to a cmnd topic. The command field identifies the operation to perform.
C2DCloud-to-Device. Message direction from the MQTT broker (or subscribing client) to the reader device. Carried on cmnd topics.
D2CDevice-to-Cloud. Message direction from the reader device to the MQTT broker. Carried on resp, event, and data topics.
Data rateThe speed at which the reader and tag exchange data, measured in kilobits per second (kbps). Determined by the link profile.
Dense readerAn environment with multiple RFID readers whose read zones overlap, causing potential interference. The DENSE_READERS profile mitigates this.
dBmDecibels relative to one milliwatt. The unit for measuring RFID transmit power and RSSI.
EPCElectronic Product Code. A unique identifier stored in a tag's EPC memory bank. The primary identifier in tag read events. Typically 96 bits (24 hex characters).
EPC Gen2The EPCglobal UHF Gen2 air interface protocol (ISO 18000-63). The communication standard between UHF RFID readers and tags. All Zebra RFID Connect readers implement Gen2.
EventAn asynchronous D2C message published to an event, data1event, or data2event topic. Events are unprompted; they are generated by device conditions, not in response to a command.
FM0A baseband encoding scheme used in the reader-to-tag link. FM0 provides the highest data rate but least noise immunity. See §17.2.2.
Firmware updateThe process of downloading and installing a new firmware image on the reader. Initiated via firmware_update. Progress reported via firmwareUpdateEVT.
HeartbeatA periodic event (heartBeatEVT) published by the reader to indicate it is online and operational. Configurable via config_events. See §10.1.
InterfaceOne of four MQTT communication domains: MGMT (management), CTRL (control), DATA (data), MDM (device management). Each maps to a separate MQTT topic prefix. See §3.2.
InventoryThe process of reading all RFID tags within the reader's RF field. Initiated with control_operation (controlType: "RFID", operation: "START").
Inventory stateOne of two states (A or B) that a tag maintains per session. The reader queries tags in a specific state to avoid re-reading tags already inventoried.
Keep-aliveThe MQTT keep-alive interval that determines how frequently the client and broker exchange PINGREQ/PINGRESP packets to maintain the connection.
KillA permanent, irreversible operation that disables an RFID tag. Requires the kill password. The tag becomes permanently non-responsive.
Kill passwordA 32-bit password stored in the tag's Reserved memory bank. Required to execute a kill operation.
Link profileA combination of encoding scheme and data rate that defines the reader-to-tag RF communication parameters. See §17.2.2.
LockAn access operation that changes the read/write permissions of a tag memory bank. Can be reversible or permanent. See §17.2.9.
Memory bankOne of four data storage areas on an EPC Gen2 tag: Reserved (passwords), EPC (identifier), TID (manufacturer data), User (application data).
Miller encodingA subcarrier encoding scheme (Miller 2, 4, or 8) used in the tag-to-reader link. Higher Miller values provide better noise immunity at the cost of data rate.
MQTTMessage Queuing Telemetry Transport. A lightweight publish/subscribe messaging protocol used by Zebra RFID Connect for all device communication. Version 3.1.1.
MQTT brokerThe central message routing server that receives published messages and distributes them to subscribers. Examples: Mosquitto, HiveMQ, AWS IoT Core. See §13.
mTLSMutual Transport Layer Security. A TLS configuration where both the client (reader) and server (broker) authenticate with certificates. See §7.
Operating modeThe RF configuration profile that determines how the reader performs inventory. Set via set_operating_mode. Profiles range from preset (e.g., BALANCED_PERFORMANCE) to fully manual (ADVANCED). See §8.1.
PC bitsProtocol Control bits. A 16-bit field in the tag's EPC memory that describes the length and structure of the EPC.
Phase angleThe phase of the tag's backscatter signal relative to the reader's transmitted signal. Used for tag location estimation in some applications.
Post-filterA software-based filter applied on the reader after tag reads, using criteria such as EPC prefix, RSSI threshold, or regex pattern. Set via set_post_filter. See §8.5.
Pre-filterA hardware-based filter applied during the Gen2 Select phase, before tag singulation. Uses memory bank contents to include or exclude tags from inventory. Configured in set_operating_mode → select. See §8.3.
Q algorithmThe adaptive algorithm that manages the Gen2 anti-collision slot count. The tagPopulation hint in ADVANCED mode seeds the Q value. See §8.2.
QoSQuality of Service. MQTT delivery guarantee level. Zebra RFID Connect uses QoS 0 (at most once) for high-frequency data and QoS 1 (at least once) for commands and responses.
Regulatory regionThe geographic region that determines the allowed RF frequency bands and power limits for the reader. Must be configured before RFID operations.
Request IDA unique string (requestId) included in every command payload. The device echoes this value in the response to enable request-response correlation.
ResponseA D2C message published to a resp topic in reply to a command. Contains a response.code and response.description. See §17.1.
Retained messageAn MQTT message stored by the broker and delivered to new subscribers immediately upon subscription. The rfid channel uses retained messages for access operation results.
RFIDRadio-Frequency Identification. A wireless technology that uses electromagnetic fields to identify and track tags attached to objects.
RSSIReceived Signal Strength Indicator. The power level of the tag's backscatter signal as measured by the reader, in dBm. Useful for proximity estimation and filtering.
Seen countThe number of times a specific tag was read during an inventory round. Reported in tag data when SEENCOUNT metadata is enabled.
SelectA Gen2 air-interface command that sets or clears a tag's inventory flag based on memory bank contents. Used to implement pre-filters. See §8.3.
SessionA Gen2 inventory parameter (S0–S3) that controls how long a tag remembers it has been inventoried. See §17.2.3.
SL flagThe Select flag on an EPC Gen2 tag. Set or cleared by Select commands. The reader can query tags based on SL flag state.
SingulationThe Gen2 process of isolating a single tag from a population for communication. The anti-collision algorithm selects one tag at a time.
TagAn RFID transponder attached to an object. Contains an integrated circuit and antenna. Stores data in memory banks (EPC, TID, User, Reserved).
Tag populationAn estimate of the number of tags expected in the reader's field. Provided as a hint to the Q algorithm for optimal anti-collision performance.
TenantAn organizational identity that scopes MQTT topics. The {tenantId} in the topic pattern isolates traffic between different deployments.
TIDTag Identification. A read-only memory bank containing the tag manufacturer's ID and chip model. Unique per tag IC.
TLSTransport Layer Security. The cryptographic protocol used to secure MQTT connections between the reader and broker. Zebra RFID Connect supports TLS 1.2 and 1.3. See §7.
Transmit powerThe RF power output of the reader's antenna, specified in 1/10 dBm units in the API (e.g., 300 = 30.0 dBm). Range varies by reader model.
User memoryAn optional memory bank on Gen2 tags available for application-defined data storage. Not all tags have user memory.
Wildcard subscriptionAn MQTT subscription using + (single-level) or # (multi-level) to match multiple topics. Used for fleet monitoring. See §3.3.
XPCExtended Protocol Control. Additional tag capability bits beyond the standard PC field. Indicates support for features like sensor data.

17.6 Changelog​

This changelog follows the Keep a Changelog convention. All dates use ISO 8601 format.

[V1.1]: 2026-01-15​

Added​

  • set_post_filter command with WHITELIST and BLACKLIST filter types for EPC-list-based filtering.
  • set_access_operation support for KILL operation type.
  • HOSTNAME and MAC fields added to tagMetaDataToEnable options.
  • mqttConnectionEVT event for monitoring MQTT broker connection state changes.
  • firmwareUpdateEVT event for tracking firmware update progress asynchronously.
  • Support for WPA3_PERSONAL and WPA3_ENTERPRISE Wi-Fi security types in set_wifi.
  • RFD9090 (IP67) added to the supported reader matrix.

Changed​

  • set_operating_mode ADVANCED profile now requires explicit linkProfile and session parameters (previously used defaults when omitted).
  • dataEVT payload now includes antennaPort field for readers with multiple antenna ports.
  • Maximum pre-filter entries increased from 4 to 8 per select array.
  • Response code 22 now overloaded: returns "Advanced config not set" in the Control domain (previously returned only "Operation in progress" in Management domain).

Fixed​

  • heartBeatEVT chargeStatus value 2 (fully charged) was not reported when the reader was connected to a cradle.
  • set_wifi with MODIFY operation no longer requires re-sending the PSK if only the ssid or securityType is changing.
  • get_operating_mode now correctly returns the select array when pre-filters are configured.
  • control_operation STOP response code changed from 0 to 12 when no operation is running, preventing false success interpretation.

[V1.0]: 2025-06-01​

Added​

  • Initial release of Zebra RFID Connect (IOTC) MQTT API.
  • 22 bidirectional commands across MGMT, CTRL, DATA, and MDM interfaces.
  • 6 asynchronous events: heartBeatEVT, alertEVT, exceptionEVT, dataEVT, barcodeEVT, mqttConnectionEVT.
  • Support for RFD40 Standard, Premium, Premium Plus, RFD40-M, and RFD9030.
  • Pre-filter (Gen2 Select) and post-filter (RSSI, PREFIX, SUFFIX, REGEX) support.
  • 6 operating mode profiles: BALANCED_PERFORMANCE, FAST_READ, CYCLE_COUNT, DENSE_READERS, OPTIMAL_BATTERY, ADVANCED.
  • Access operations: READ, WRITE, LOCK.
  • Wi-Fi configuration with WPA2_PERSONAL, WPA2_ENTERPRISE, and OPEN security types.
  • TLS and mTLS support for MQTT broker connections.
  • OTA firmware update via firmware_update command.

Security​

  • All MQTT connections require TLS 1.2 or higher by default.
  • Certificate provisioning via set_certificate for broker, Wi-Fi, and client authentication.

[V0.9-beta]: 2025-03-01​

Added​

  • Beta release for early adopter program.
  • Core MQTT topic hierarchy and 16 initial commands.
  • heartBeatEVT and alertEVT events.
  • Support for RFD40 Premium with firmware 3.8.x.

Changed​

  • Topic hierarchy changed from {tenantId}/clients/{interface}/{channel}/{deviceSerial} to {tenantId}/{interface}/clients/{channel}/{deviceSerial} (interface moved before clients).

Fixed​

  • MQTT reconnection loop when broker certificate was rotated without updating the reader.

Security​

  • Added cleanSession parameter to config_endpoint to prevent stale session data on reconnection.
